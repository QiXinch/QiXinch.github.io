Movtivation

在程序中，一种经常出现的情况是读取命令行参数或这是配置文件，在程序运行时根据参数来做出不同的行为，而参数在程序运行过程中一般是不变的。

在解析出flag之后，根据flag来做出相应的行为。当然我们可以使用likely和unlikely来修饰flag,帮助编译器来优化
在x86_64架构中，使用gcc 14.2 -O3 -std=c++17编译会得到：

尽管flag的值是不变的，但是每次仍然需要读取值，进行判断，如果这种情况出现的次数较多而且被频繁执行，仍然会对程序的性能造成影响，如在linux tracepoint中，随着tracepoint数量的增加，这种情况会对程序的性能造成影响，所以需要尽可能减少这种影响。

由于flag的值基本上是不变的，因此在flag的值为真时，我们可以使用nop替换mov, test, jne，在flag的值为假时，使用jmp .L8替换mov, test, jne,这样就避免了读取和分支判断。

这就是static_key所要实现的功能：动态地修改指令，避免分支跳转指令。

API

API只是为了说明，没有和linux代码完全对应

一个使用static_key的例子

为了使用static_key，首先需要定义一个全局变量key(不能在栈上或堆上，后面会解释)，key默认可以为True或False，在判断时使用static_branch_likely(key)或者是static_branch_unlikely(key),如果在执行过程中需要改变key的值，可以使用static_key_enable和static_key_disable设置key的值为True或False,最后在使用之前需要调用static_key_init来进行初始化。

Implement

使用c++实现，只是一种实现，和linux内核实现并不完全对应，也不是最高效的

Data struct

static_key相当于使用的flag,不过包含了其他的信息. ebabled记录当前的flag是True还是False. entries记录了使用当前flag的代码的信息，在开启或关闭flag的时候需要对这些地方进行修改。

jump_entry记录的是使用flag的代码的信息，code代表了需要修改为nop或jmp的代码的地址，target代表了if判断的跳转地址，在修改为jmp的时候需要用到。key代表了使用flag的地址，由于key是8 byte对齐的，因此使用key的最低一位是否为1来表示当前代码使用的是static_branch_likely还是static_branch_unlikely

Function

DEFINE_STATIC_KEY_{TRUE, FALSE}

static_branch_{likely, unlikely}

这一段代码根据linux内核static_key的相关实现修改而来，代码实现得相当巧妙。

首先分析static_key_branch, static_key_branch真正起作用的代码只有jmp %l[l_yes]，即跳到l_yes，而剩余的则是用来构造jump_entry, 其中表示的位置都是相对于当前位置的偏移，构造的jump_entry会被放到__jump_entry段里面(这也是static_key变量为什么不能是栈或堆变量的原因，因为需要取地址)。

对于static_branch_likely，可以看到其最终返回的是False，而对于static_branch_unlikely最终返回的是True,这样设置最终达成的效果是

可以看到当static_key是True，并且是likely的话，只需要一个nop就可以实现了。虽然不知道为什么会生成这样，但是如果反过来设置，将会达成相反的效果，即当static_key是True，并且是likely的话，这样会需要一个jmp指令，显然不如nop效率更高😄。

我们将所以的jump_entry放到了__jump_table段，为了使用这个段，我们需要修改链接脚本，在里面加入

static_key_{enable, disable}

这里只列出了staitc_key_enable的实现，static_key_disable的实现类似。

enabled代表是否启用flag,记likely代表是否是likely，则enabled ^ likey = 1 => jmp, enabled ^ likely = 0 => nop

static_key_init

static_key_init将属于相同key的jump_entry合并到一起，方便后面的修改。

Analysis

static_key适用于判断条件很少改变的情况，因为enable或disable staitic的代价还是很大的，需要相应的jump_entry全部修改一遍。

动态修改代码部分和架构以及操作系统有关，在linux系统，x86_64架构下，首先需要使用mprotect使得相应的地址可写，由于x86_64是变长指令集，所以需要分析指令的长度，做出相应的修改(nop指令的长度范围为1-9byte,而jmp的长度可以为2, 5, 6， 在修改时长度需要对应，所以在填充指令时使用的是jmp，但是感觉可以优化static_key_init函数，在编译结束后，扫描elf文件，得到__jump_entry段和所有的static_key变量，这时我们可以静态分析，修改对应的指令)。
