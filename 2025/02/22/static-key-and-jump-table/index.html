Movtivation

在程序中，一种经常出现的情况是读取命令行参数或这是配置文件，在程序运行时根据参数来做出不同的行为，而参数在程序运行过程中一般是不变的。

在解析出flag之后，根据flag来做出相应的行为。当然我们可以使用likely和unlikely来修饰flag,帮助编译器来优化
在x86_64架构中，使用gcc 14.2 -O3 -std=c++17编译会得到：

尽管flag的值是不变的，但是每次仍然需要读取值，进行判断，如果这种情况出现的次数较多而且被频繁执行，仍然会对程序的性能造成影响，如在linux tracepoint中，随着tracepoint数量的增加，这种情况会对程序的性能造成影响，所以需要尽可能减少这种影响。

由于flag的值基本上是不变的，因此在flag的值为真时，我们可以使用nop替换mov, test, jne，在flag的值为假时，使用jmp .L8替换mov, test, jne,这样就避免了读取和分支判断。

这就是static_key所要实现的功能：动态地修改指令，避免分支跳转指令。

API

API只是为了说明，没有和linux代码完全对应

一个使用static_key的例子

为了使用static_key，首先需要定义一个全局变量key(不能在栈上或堆上，后面会解释)，key默认可以为True或False，在判断时使用static_branch_likely(key)或者是static_branch_unlikely(key),如果在执行过程中需要改变key的值，可以使用static_key_enable和static_key_disable设置key的值为True或False,最后在使用之前需要调用static_key_init来进行初始化。

Implement

使用c++实现，只是一种实现，和linux内核实现并不完全对应，也不是最高效的

Data struct

static_key相当于使用的flag,不过包含了其他的信息. ebabled记录当前的flag是True还是False. entries记录了使用当前flag的代码的信息，在开启或关闭flag的时候需要对这些地方进行修改。

jump_entry记录的是使用flag的代码的信息，code代表了需要修改为nop或jmp的代码的地址，target代表了if判断的跳转地址，在修改为jmp的时候需要用到。key代表了使用flag的地址，由于key是8 byte对齐的，因此使用key的最低一位是否为1来表示当前代码使用的是static_branch_likely还是static_branch_unlikely

Function
